                      │

## 核心模块说明

### 1. OCR + 切片模块

**核心文件**: `backend/databases/ingestion/indexing/pipeline.py`

**功能**:
- OCR 识别：使用 MinerU 解析 PDF（支持文本、图片、表格、公式）
- 文档切片：基于层级结构分割（标题、段落、表格）
- 图片提取：从 Markdown 提取图片引用 (NEW! 2025-12-19)
- VLM 语义化：为图片生成语义描述 (qwen3-vl-plus)
- 向量化：生成文本嵌入（BGE-M3 模型）
- 数据入库：写入 MongoDB + Milvus

**关键类**: `DocumentIngestionPipeline`

### 2. 知识图谱构建模块

**核心文件**: `backend/databases/graph/builders/kg_builder.py`

**功能**:
- 实体抽取：从文本中识别医疗建筑相关实体
- 关系抽取：识别实体间的语义关系
- 质量控制：置信度过滤、去重、规范化
- 双向链接：chunk_id ↔ entity_id 关联
- 图谱构建：写入 Neo4j 数据库

**关键类**: `MedicalKGBuilder`

### 3. 统一 CLI 工具

**核心文件**: `backend/cli/`

- `rebuild.py`: 全量重建管理
- `build_kg.py`: 知识图谱构建
- `batch_indexer.py`: 批量文档处理

## 全量重建流程

### 场景
- 首次部署系统
- 代码逻辑重大变更（如切片策略、Schema 调整）
- 数据库损坏需要重建

### 完整步骤

#### Step 1: 清空现有数据库

```bash
# 方法1: 使用统一的重建工具（推荐）
python -m backend.cli.rebuild --clear-only

# 方法2: 手动清空各数据库
# MongoDB
mongosh mongodb://admin:mediarch2024@localhost:27017/mediarch --eval "
  db.mediarch_documents.deleteMany({});
  db.mediarch_chunks.deleteMany({});
"

# Neo4j（保留预注入的概念节点）
cypher-shell -u neo4j -p mediarch2024 "
  MATCH (n) WHERE NOT n:Concept DELETE n;
  MATCH ()-[r]-() DELETE r;
"

# Milvus（通过 Python API）
python -c "
from pymilvus import connections, utility
connections.connect(host='localhost', port='19530')
utility.drop_collection('mediarch_chunks')
"
```

**参数说明**:
- `--clear-only`: 仅清空，不执行重建
- `--clear-all`: 包括清空 Neo4j 的预注入概念节点
- `--skip-neo4j`: 跳过 Neo4j 清空

#### Step 2: 执行文档 OCR + 向量化入库

```bash
# 方法1: 使用统一 CLI（推荐，包含完整流程）
python -m backend.cli.rebuild

# 方法2: 使用独立的 ingestion 脚本（旧方式）
python backend/databases/ingestion/run_poc.py

# 方法3: 仅重建索引，不清空（断点续传）
python -m backend.cli.rebuild --no-clear
```

**执行内容**:
1. 扫描 `backend/databases/documents/` 下的所有 PDF
2. 对每个文档执行：
   - MinerU OCR 解析
   - Markdown + JSON 输出保存到 `documents_ocr/`
   - 文档切片（chunking）
   - 图片提取 + VLM 语义化（如启用）
   - 文本嵌入生成（BGE-M3）
   - 写入 MongoDB (`mediarch_documents`, `mediarch_chunks`)
   - 写入 Milvus (`mediarch_chunks` collection)

**预期输出**:
- MongoDB: ~20 documents, ~1,900 chunks (含图片)
- Milvus: ~1,900 vectors (768维)
- 日志文件: `rebuild_YYYYMMDD_HHMMSS.log`

**耗时估算**: 2-4 小时（取决于文档数量和 VLM 启用状态）

#### Step 3: 构建知识图谱

```bash
# 方法1: 使用统一 CLI（推荐）
python -m backend.cli.build_kg

# 方法2: 使用独立脚本（旧方式）
python backend/databases/graph/build_kg_with_deepseek.py

# 选项: 跳过磁盘检查（临时目录空间不足时）
python -m backend.cli.build_kg --skip-disk-check

# 选项: 禁用 Milvus 实体向量索引
python -m backend.cli.build_kg --no-milvus
```

**执行内容**:
1. 连接 MongoDB，读取所有 `mediarch_chunks`
2. 分批调用 DeepSeek V3 LLM：
   - 实体抽取（Entity Extraction）
   - 关系抽取（Relation Extraction）
   - 质量评分（confidence scoring）
3. 质量控制：过滤低置信度结果、去重
4. 写入 Neo4j：
   - 创建节点（实体）
   - 创建关系（带 chunk_id 追踪）
5. 更新 MongoDB chunks：添加 `entity_ids` 字段
6. （可选）构建 Milvus 实体属性向量索引

**预期输出**:
- Neo4j: ~10,000 节点, ~30,000 关系
- MongoDB chunks: 90%+ 包含 `entity_ids`
- 成本: $0.50 - $1.50 USD (DeepSeek API)

**耗时估算**: 3-6 小时（取决于 chunk 数量）

#### Step 4: 验证数据完整性

```bash
# 使用重建工具的验证功能
python -m backend.cli.rebuild --verify-only

# 手动验证（Python）
python -c "
from pymongo import MongoClient
from pymilvus import connections, Collection
from neo4j import GraphDatabase

# MongoDB
client = MongoClient('mongodb://admin:mediarch2024@localhost:27017/')
db = client['mediarch']
print(f'Documents: {db.mediarch_documents.count_documents({})}')
print(f'Chunks: {db.mediarch_chunks.count_documents({})}')

# Milvus
connections.connect(host='localhost', port='19530')
coll = Collection('mediarch_chunks')
print(f'Vectors: {coll.num_entities}')

# Neo4j
driver = GraphDatabase.driver('bolt://localhost:7687', auth=('neo4j', 'mediarch2024'))
with driver.session() as session:
    result = session.run('MATCH (n) RETURN count(n) as nodes')
    print(f'Neo4j Nodes: {result.single()[\"nodes\"]}')
"
```

**完整流程总耗时**: 6-10 小时

## 增量更新流程

### 场景
- 添加新的 PDF 文档
- 更新现有文档内容
- 定期维护和优化

### 核心机制

**文档版本控制**: 使用 `doc_key` (文件路径) 作为唯一标识

**增量更新策略**:
1. **文档级别**:
   - 检查 MongoDB 中是否存在该 `doc_key`
   - 存在 → 删除旧 chunks + 旧向量 + 旧图谱关系
   - 重新处理该文档

2. **知识图谱级别**:
   - 保留未被更新文档引用的实体和关系
   - 更新受影响的节点和边

### 增量更新步骤

#### Step 1: 添加新文档

```bash
# 1. 将新 PDF 放到文档目录
cp /path/to/new.pdf backend/databases/documents/标准规范/

# 2. 执行增量索引（自动检测新文档）
python -m backend.cli.batch_indexer --incremental

# 或使用 run_poc.py（设置环境变量）
export FORCE_REINGEST=1  # 强制重新处理已存在的文档
python backend/databases/ingestion/run_poc.py
```

**`FORCE_REINGEST` 环境变量**:
- `FORCE_REINGEST=0` (默认): 跳过已处理的文档
- `FORCE_REINGEST=1`: 强制重新处理所有文档（用于代码更新后）

#### Step 2: 增量更新知识图谱

```bash
# 方法1: 仅处理新 chunks（推荐）
python -m backend.cli.build_kg --incremental

# 方法2: 全量重建知识图谱（耗时长，但最准确）
python -m backend.cli.build_kg
```

**增量模式行为**:
- 读取 MongoDB 中 `entity_ids` 为空或需要更新的 chunks
- 仅对这些 chunks 执行实体抽取
- 更新 Neo4j 图谱（新增节点和关系）
- 避免重复处理已有的实体

#### Step 3: 验证增量更新

```python
# 检查新文档是否被索引
python -c "
from pymongo import MongoClient
client = MongoClient('mongodb://admin:mediarch2024@localhost:27017/')
db = client['mediarch']

# 查看最新的文档
latest = db.mediarch_documents.find().sort('_id', -1).limit(1)
for doc in latest:
    print(f'Latest doc: {doc[\"title\"]}')
    print(f'Chunks count: {db.mediarch_chunks.count_documents({\"source_document\": doc[\"title\"]})}')
"
```

### 增量更新最佳实践

1. **定期执行**: 建议每周执行一次增量更新
2. **备份数据**: 在大规模更新前备份 Neo4j 数据
3. **监控成本**: 增量更新仍会调用 LLM API，注意成本控制
4. **验证结果**: 每次更新后验证数据一致性

## 常见问题

### Q1: 全量重建需要多长时间？

**回答**:
- OCR + 向量化: 2-4 小时（20个文档，含VLM）
- 知识图谱构建: 3-6 小时（~2000 chunks）
- **总计**: 6-10 小时

**优化建议**:
- 禁用 VLM 可节省 30-50% 时间
- 使用更快的 OCR 引擎（如 Azure OCR）
- 增加 LLM 并发请求（需注意 API 限流）

### Q2: 增量更新是否会影响已有数据？

**回答**:
- **MongoDB/Milvus**: 仅更新/添加新文档的 chunks
- **Neo4j**: 增量模式会新增节点和关系，不会删除旧数据
- **风险**: 如果文档被删除，旧的 chunks 和实体不会自动清理

**建议**: 定期执行全量重建（如每月一次）

### Q3: 如何控制 LLM API 成本？

**回答**:
- 使用 `DEEPSEEK_MAX_CHUNKS` 环境变量限制处理的 chunk 数量
- 调整 `confidence_threshold` 减少低质量抽取
- 使用缓存机制避免重复处理

示例:
```bash
export DEEPSEEK_MAX_CHUNKS=500  # 仅处理前 500 个 chunks
export KG_CONFIDENCE_THRESHOLD=0.8  # 提高置信度阈值
python -m backend.cli.build_kg
```

### Q4: VLM 处理失败怎么办？

**回答**:
- 检查环境变量: `VLM_API_KEY`, `VLM_BASE_URL`
- 查看日志: `backend/logs/pipeline.log`
- 临时禁用 VLM: `VLM_ENABLED=0`
- 图片仍会被索引，但缺少语义描述

### Q5: 如何回滚到之前的数据状态？

**回答**:
- **Neo4j**: 使用 `neo4j-admin dump/load` 备份恢复
- **MongoDB**: 使用 `mongodump/mongorestore`
- **Milvus**: 暂不支持增量备份，建议定期全量导出

备份命令:
```bash
# Neo4j
neo4j-admin dump --database=neo4j --to=/backup/neo4j-backup.dump

# MongoDB
mongodump --uri="mongodb://admin:mediarch2024@localhost:27017/mediarch" --out=/backup/mongo
```

### Q6: 为什么图片 chunks 数量为 0？

**回答**:
- 2025-12-19 之前的版本未实现图片提取功能
- 需要执行全量重建以提取图片
- 确保 `VLM_ENABLED=1` 且 API key 配置正确

### Q7: 如何查看构建进度？

**回答**:
- 所有脚本都支持实时进度显示（Rich UI）
- 查看日志文件: `rebuild_*.log`, `kg_build_*.log`
- MongoDB 中间状态查询:
  ```python
  db.mediarch_chunks.count_documents({"entity_ids": {"$exists": True}})
  ```

## 附录

### A. 环境变量参考

| 变量名 | 默认值 | 说明 |
|--------|--------|------|
| `FORCE_REINGEST` | 0 | 1=强制重新处理已存在文档 |
| `VLM_ENABLED` | 1 | 1=启用VLM图片语义化 |
| `VLM_API_KEY` | - | VLM API密钥 |
| `DEEPSEEK_API_KEY` | - | DeepSeek LLM密钥 |
| `DEEPSEEK_MAX_CHUNKS` | -1 | 限制处理chunk数量（-1=全部） |
| `KG_CONFIDENCE_THRESHOLD` | 0.6 | 实体抽取置信度阈值 |
| `SKIP_DISK_CHECK` | 0 | 1=跳过磁盘空间检查 |
| `KEEP_MINERU_OUTPUTS` | 1 | 1=保留OCR临时文件 |

### B. 数据库 Schema 参考

**MongoDB Chunk 结构**:
```json
{
  "chunk_id": "uuid",
  "content": "文本内容",
  "content_type": "text|image|table",
  "image_url": "images/xxx.jpg",
  "image_url_abs": "/absolute/path/xxx.jpg",
  "embedding": [0.1, 0.2, ...],
  "entity_ids": ["entity_1", "entity_2"],
  "source_document": "文档名称.pdf",
  "metadata": {
    "page_number": 1,
    "section": "章节名称",
    "vlm_processed": true
  }
}
```

**Neo4j 节点结构**:
```cypher
(:Entity {
  id: "entity_123",
  name: "实体名称",
  type: "实体类型",
  chunk_ids: ["chunk_1", "chunk_2"]
})
```

**Neo4j 关系结构**:
```cypher
(:Entity)-[:RELATION_TYPE {
  chunk_id: "chunk_1",
  confidence: 0.85,
  relation: "关系描述"
}]->(:Entity)
```
