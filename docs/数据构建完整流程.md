## 系统架构概览

### 数据库组件

```
┌─────────────────────────────────────────────────────────────┐
│                    MediArch 数据架构                          │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  📄 PDF 文档                                                  │
│      ↓                                                       │
│  🔍 OCR 处理 (MinerU/TextIn)                                 │
│      ↓                                                       │
│  ┌──────────────┬──────────────┬──────────────┐            │
│  │              │              │              │            │
│  ▼              ▼              ▼              ▼            │
│  MongoDB      Milvus        Neo4j         本地图片          │
│  (文档/Chunks)  (向量)      (知识图谱)     (静态资源)        │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

### 核心模块

| 模块 | 功能 | 文件路径 |
|------|------|---------|
| **OCR处理** | PDF解析、图片提取、VLM描述 | `backend/databases/ingestion/` |
| **向量化** | 文本嵌入、向量存储 | `backend/databases/ingestion/indexing/` |
| **知识图谱** | 实体抽取、关系构建 | `backend/databases/graph/` |
| **CLI工具** | 命令行接口 | `backend/cli/` |

## 全量重建流程

### 适用场景
- 首次部署系统
- 数据损坏需要重建
- 更换OCR引擎或重大架构升级
- 清理所有旧数据重新开始

### 🔄 完整流程图

```
[开始]
  ↓
[步骤1] 清空所有数据库
  ├─ 清空 MongoDB (documents, mediarch_chunks)
  ├─ 清空 Milvus (mediarch_chunks collection)
  └─ 清空 Neo4j (可选保留概念节点)
  ↓
[步骤2] OCR + 向量化 (自动完成)
  ├─ PDF 文档 OCR 解析
  ├─ 提取文本和图片
  ├─ VLM 生成图片描述
  ├─ 文本分块 (chunking)
  ├─ 向量嵌入 (embedding)
  ├─ 写入 MongoDB (documents + chunks)
  └─ 写入 Milvus (向量索引)
  ↓
[步骤3] 预注入空间知识 (可选)
  └─ 注入医疗建筑概念骨架
  ↓
[步骤4] 构建知识图谱
  ├─ 从 MongoDB 读取 chunks
  ├─ 使用 DeepSeek V3 抽取实体和关系
  └─ 写入 Neo4j
  ↓
[步骤5] 验证数据完整性
  └─ 检查各数据库状态
  ↓
[完成]
```

### 📝 详细步骤

#### 步骤1: 清空并重建向量数据（推荐使用新工具）

```bash
# 方法A: 使用新的统一重建工具（推荐）
python -m backend.cli.rebuild --clear-all

# 说明：
# - 清空所有数据库（MongoDB + Milvus + Neo4j）
# - 自动完成 OCR + 向量化
# - 支持断点续传
# - 详细的进度显示和日志
# - 自动验证结果

# 可选参数:
python -m backend.cli.rebuild --clear-all --verbose  # 详细日志
python -m backend.cli.rebuild --skip-neo4j           # 保留Neo4j数据
```

**或**

```bash
# 方法B: 使用原有的交互式工具
python backend/databases/ingestion/run_poc.py

# 说明：
# - 交互式界面
# - 支持轻录入模式（每次3页）
# - 支持补齐模式（一次性完成）
# - 断点续传，增量处理
# - 生成 OCR 进度报告
```

**方法A vs 方法B 对比:**

| 特性 | 方法A (rebuild) | 方法B (run_poc) |
|------|----------------|----------------|
| 界面 | 自动化 | 交互式 |
| 清空数据库 | 自动 | 需手动 |
| 进度跟踪 | 基础 | 详细（页级） |
| 断点续传 | 支持 | 支持 |
| 适用场景 | 全量重建 | 增量处理 |

#### 步骤2: 预注入空间知识（可选）

```bash
# 注入医疗建筑概念骨架（约150个空间节点）
python backend/databases/graph/utils/seed_ontology_v2.py

# 说明：
# - 创建医院、部门、功能分区、标准空间等概念节点
# - 这些节点会被标记为 is_concept=true
# - 可以在重建时选择保留（--clear-all 会删除）
```

#### 步骤3: 构建知识图谱

```bash
# 方法A: 使用 CLI 工具（推荐）
python -m backend.cli.build_kg

# 方法B: 直接运行原始脚本
python backend/databases/graph/build_kg_with_deepseek.py

# 说明：
# - 从 MongoDB 读取所有 chunks
# - 使用 DeepSeek V3 抽取实体和关系
# - 自动去重和合并节点
# - 写入 Neo4j
# - 显示成本估算和进度
```

#### 步骤4: 验证结果

```bash
# 验证所有数据库状态
python scripts/verify_databases.py

# 查看统计信息
python scripts/251219test_rebuild_service.py
```

### ⏱️ 预计耗时

假设有 20 个 PDF 文档，平均每个 50 页：

| 步骤 | 耗时 | 说明 |
|------|------|------|
| 清空数据库 | ~1分钟 | 取决于数据量 |
| OCR + 向量化 | 30-90分钟 | 取决于文档大小和服务器性能 |
| 预注入空间知识 | ~30秒 | 固定时间 |
| 构建知识图谱 | 20-40分钟 | 取决于 chunk 数量和 API 速度 |
| **总计** | **~1-2.5小时** | |

## 增量更新流程

### 适用场景
- 添加新的 PDF 文档
- 更新已有文档内容
- 修复个别文档的处理错误

### 🔄 增量流程图

```
[新 PDF 文档]
  ↓
[步骤1] OCR + 向量化（自动跳过已存在）
  ├─ 系统检查 document_id
  ├─ 已存在 → 跳过
  └─ 不存在 → 处理
  ↓
[步骤2] 增量构建知识图谱（可选）
  └─ 只处理新文档的 chunks
  ↓
[完成]
```

### 📝 详细步骤

#### 步骤1: 添加新文档到目录

```bash
# 将新 PDF 放入对应分类目录
cp new_document.pdf backend/databases/documents/标准规范/
```

#### 步骤2: 增量 OCR + 向量化

```bash
# 方法A: 使用 batch_indexer（推荐）
python -m backend.cli.batch_indexer

# 说明：
# - 自动扫描所有分类目录
# - 检测新文档（基于 document_id）
# - 只处理新文档，跳过已存在的
# - 不需要设置 FORCE_REINGEST

# 输出示例：
# [1/21] 处理: existing_doc.pdf
# [SKIP] 文档已存在，跳过
# [21/21] 处理: new_document.pdf
# [OK] 成功处理 ✓
```

**或**

```bash
# 方法B: 使用 run_poc.py 交互模式
python backend/databases/ingestion/run_poc.py

# 选择 "补齐全部缺口" 选项
# 系统会自动处理新文档
```

**强制重新处理已有文档（慎用）:**

```bash
# 设置环境变量强制重新处理
export FORCE_REINGEST=1
python -m backend.cli.batch_indexer

# Windows:
set FORCE_REINGEST=1
python -m backend.cli.batch_indexer
```

#### 步骤3: 增量构建知识图谱（可选）

```bash
# 方法A: 处理所有新 chunks（推荐）
python -m backend.cli.build_kg

# 说明：
# - 知识图谱构建器会自动去重
# - 相同实体会被合并
# - 新关系会被添加

# 方法B: 使用增量更新脚本（需要完善）
python backend/databases/graph/utils/incremental_update.py --auto

# 注意：此脚本仍在开发中，建议使用方法A
```

### ⏱️ 预计耗时

增量添加 1 个新文档（50页）：

| 步骤 | 耗时 | 说明 |
|------|------|------|
| OCR + 向量化 | 3-5分钟 | 单个文档 |
| 构建知识图谱 | 2-5分钟 | 只处理新chunks |
| **总计** | **~5-10分钟** | |

## 核心概念说明

### 1. 幂等性设计

系统通过 `document_id` 保证幂等性：
- `document_id` = hash(文件路径 + 文件哈希)
- MongoDB 中 `document_id` 字段有唯一索引
- 重复处理同一文档会被自动跳过

### 2. 强制重新处理

设置环境变量 `FORCE_REINGEST=1` 可以：
- 跳过幂等性检查
- 删除旧数据
- 重新处理文档

### 3. 断点续传

`run_poc.py` 通过进度跟踪器实现：
- 记录每个文档的处理进度（页级）
- 自动检测缺口
- 支持中断后继续

### 4. 知识图谱去重

构建器会自动：
- 合并相同名称的实体节点
- 去重重复的关系
- 保留 `is_concept=true` 的概念节点

## 故障排查

### 问题1: 向量维度不匹配

```
错误: Milvus schema mismatch - vector dimension expected 3072, got 1536
```

**解决方案:**
```bash
# 删除 Milvus collection 重建
python -m backend.cli.rebuild --skip-mongodb --skip-neo4j
```

### 问题2: MongoDB 重复键错误

```
错误: E11000 duplicate key error collection: mediarch.documents index: document_id_unique
```

**解决方案:**
```bash
# 设置强制重新处理
set FORCE_REINGEST=1
python -m backend.cli.batch_indexer
```

### 问题3: Neo4j 节点重复

```
警告: Found duplicate nodes for entity "手术室"
```

**解决方案:**
```bash
# 运行去重脚本
python backend/databases/graph/optimization/clean_existing_graph.py

# 或完全重建
python -m backend.cli.rebuild --skip-mongodb --skip-milvus
```

### 问题4: OCR 进度卡住

**解决方案:**
```bash
# 查看进度
python backend/databases/ingestion/run_poc.py
# 选择 "查看当前进度"

# 清除特定文档的进度记录
# 编辑: backend/databases/ingestion/ocr_res/ocr_progress.json
# 删除对应文档的记录
```

### 问题5: 磁盘空间不足

```
错误: No space left on device
```

**解决方案:**
```bash
# 设置临时目录到其他盘
set TMPDIR=E:\temp
set SKIP_DISK_CHECK=1

# 清理 Docker 卷（如果使用 Docker）
docker system prune -a
```

## 数据流向图

```
PDF 文档
   ↓
OCR (MinerU) ────────┐
   ↓                 │
文本 chunks          │ 图片文件
   ↓                 ↓
Embedding        VLM 描述
   ↓                 ↓
MongoDB ←────── (合并)
   ↓
Milvus (向量索引)
   ↓
Neo4j (知识图谱)
   ↑
   └─ DeepSeek V3 (实体抽取)
```

## 环境变量配置

重要的环境变量（.env 文件）：

```bash
# OCR 引擎
OCR_ENGINE=mineru                    # 使用 MinerU
MINERU_BACKEND=pipeline              # MinerU 后端
MINERU_USE_CUDA=1                    # 启用 GPU

# 强制重新处理
FORCE_REINGEST=0                     # 默认跳过已存在文档

# MongoDB
MONGODB_URI=mongodb://localhost:27017
MONGODB_DATABASE=mediarch

# Milvus
MILVUS_HOST=localhost
MILVUS_PORT=19530

# Neo4j
NEO4J_URI=bolt://localhost:7687
NEO4J_USER=neo4j
NEO4J_PASSWORD=mediarch2024
NEO4J_DATABASE=neo4j

# DeepSeek API
DEEPSEEK_API_KEY=your_api_key
DEEPSEEK_BASE_URL=https://api.deepseek.com

# 临时目录（解决磁盘空间问题）
TMPDIR=E:\temp
SKIP_DISK_CHECK=1
```
