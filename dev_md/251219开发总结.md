### 问题解决记录

#### 问题1：数据库重建后图片 chunks 数量为 0
- **影响范围**：整个检索系统无法返回图片相关内容，知识图谱无法关联图片信息
- **根本原因**：
  1. `chunking.py` 原逻辑只在 `details[]` 数组中查找图片，但 MinerU OCR 不在此数组中保存图片引用
  2. 图片实际以 Markdown 语法 `![](images/xxx.jpg)` 存储在 `result.markdown` 字段中
  3. 代码跳过了空文本的 detail 项，导致图片被忽略
- **解决方案**：
  1. 在处理 `details[]` 之前，先从完整 Markdown 文本提取所有图片
  2. 为每张图片创建独立的 image chunk，设置正确的 `image_url_abs` 路径
  3. 路径构建：`(artifacts_dir_parent / "images" / filename).resolve()`
- **经验总结**：
  - OCR 工具的输出结构差异很大，不能假设所有内容都在 `details[]` 中
  - 应优先使用最完整的数据源（Markdown 全文）而非结构化的局部数据

#### 问题2：VLM 处理失败，路径显示 `mineru_outputs/images/xxx.jpg`
- **影响范围**：虽然图片 chunks 被创建，但 VLM 无法读取图片生成描述
- **根本原因**：
  - `pipeline.py` 中的旧代码（493-506行）在 chunking 之后覆写了 `image_url_abs`
  - 旧代码从 `path_map` 获取路径，但 `path_map` 的键是 MinerU 输出的临时路径（包含 `mineru_outputs/`）
  - chunking.py 已设置正确路径，但被 pipeline.py 的后处理逻辑覆盖
- **解决方案**：
  - 在 `pipeline.py` 500行添加条件：`if ch.get('content_type') == 'image' and not ch.get('image_url_abs')`
  - 仅在 `image_url_abs` 未设置时才从 `path_map` 补充（向后兼容）
  - 保护 chunking.py 设置的正确路径不被覆盖
- **经验总结**：
  - 数据流经多个处理阶段时，要明确每个阶段的职责边界
  - 后续阶段不应假设前置阶段的处理不完整
  - 添加防御性检查（`if not existing_value`）避免误覆盖

## 💡 技术亮点与收获

### 关键决策

1. **Markdown 优先策略**
   - 决策：不依赖 OCR 工具的结构化输出（`details[]`），直接解析完整 Markdown 文本
   - 理由：Markdown 是最完整的输出，包含所有内容元素
   - 好处：适配性更强，不受 OCR 工具输出格式变化影响

2. **VLM 集成时机**
   - 决策：在 chunking 完成后、向量化之前进行 VLM 处理
   - 理由：VLM 生成的语义描述需要被向量化才能参与检索
   - 好处：图片也能通过语义相似度被检索到

### 技术积累

**图片路径构建模式**（可复用）：
```python
from pathlib import Path

# 场景：OCR 输出目录在文档目录的子目录中，图片在文档根目录
artifacts_dir = ".../doc/mineru_outputs"  # OCR 临时目录
artifacts_path = Path(artifacts_dir)
doc_dir = artifacts_path.parent  # 回到文档根目录
image_abs_path = (doc_dir / "images" / filename).resolve()  # 标准化路径

# 检查文件存在性（resolve() 后的路径更可靠）
if image_abs_path.exists():
    process_image(str(image_abs_path))
```

**正则提取 Markdown 图片**（可复用）：
```python
import re

markdown_text = "文本内容 ![图片标题](images/abc.jpg) 更多文本"
pattern = r'!\[([^\]]*)\]\(images/([^)]+)\)'
matches = re.findall(pattern, markdown_text)
# matches = [('图片标题', 'abc.jpg')]

for caption, filename in matches:
    print(f"发现图片: {filename}, 标题: {caption}")
```

### 踩坑记录

1. **Path.resolve() 的重要性**
   - 问题：使用 `Path(...)` 构建的相对路径在 `.exists()` 检查时不可靠
   - 解决：始终使用 `.resolve()` 转为绝对路径再进行文件操作
   - 原因：相对路径的工作目录可能与预期不一致

2. **数据覆盖的隐患**
   - 问题：多个处理阶段对同一字段赋值，后续阶段可能覆盖正确值
   - 解决：每个阶段只处理自己负责的数据，使用条件检查避免误覆盖
   - 示例：`if not existing_value: new_value = ...`

3. **环境变量命名一致性**
   - 注意：项目中 VLM 相关的环境变量从 `KG_VISION_*` 改为 `VLM_*`
   - 确保 `.env` 文件与代码中使用的变量名一致
