3. **建立“局部→整体”的可控入库流程**：先单 PDF、再扩页段、再按类别、最后全量；同时避免重复向量污染（FORCE_REINGEST 时清理 Mongo + Milvus）。

## 🧭 背景问题（今天主要在解决什么）

### 1) 之前为什么“仍然没有显示图片”
- 数据库里其实已经有 `content_type="image"` 的 chunks，且部分 chunk 带 `image_url`。  
- 但对话 API 在返回时：
  - **没有把图片 URL 列表统一收集到 `images[]`**（或未在 `message` 中给前端可识别的占位符）。  
  - **引用（citations）可能被 `max_citations` 截断**，导致图片引用被挤掉，即使库里有图也“不显示”。  

### 2) 之前为什么“图片检索不准/总是无标题”
- 图片 chunk 的 `content` 若是 `[图片: 无标题]`，即使入库，也很难通过语义向量命中正确图片。  
- 需要 VLM 把图片生成“可检索描述”（例如“负压病房平面示例：患者走廊/医护走廊/传递窗/更衣淋浴…）”，并写回 `content` 后再 embedding。  

### 3) 为什么不建议一上来“重建整个 Mongo/Milvus”
- 全库重建成本高、风险大、验证周期长。  
- 更好的方式是：**单 PDF → 扩大页段 → 类别批量 → 全量**，每一步都用接口测试确认效果。  
- 为了支持这种方式，需要 FORCE_REINGEST 时同时清理 Mongo + Milvus 的旧数据，避免重复向量污染。

**目标**：无论是非流式还是流式，对话响应都能带 `images[]`，并且 `message` 带占位符。  

实现点：

**目标**：用户明确要图时，即使检索命中是文本，也能补回同文档附近图片，提升体验与可见性。

- `_want_images(query)` 判断是否“要图”
- `_collect_doc_page_hints(chunks)` 从命中 chunks 提取 (doc_id, page hints)
- `retriever.get_image_chunks_near_pages(doc_id, pages, per_doc, window)` 补图

涉及文件：
- `backend/app/agents/mongodb_agent/agent.py`

### 3) MongoDB 检索器：按页码窗口补图
新增方法：
- `MongoDBChunkRetriever.get_image_chunks_near_pages(doc_id, near_pages, limit, page_window)`

并修复一个今天发现的插入错误：
- `_search_with_text_index()` 原本因为插入新方法时位置不当，导致正文逻辑被挤到“return 之后不可达”。  
- 已把 text index 搜索逻辑恢复到 `_search_with_text_index()`，保证“文本索引策略”仍可用。

涉及文件：
- `backend/app/services/mongodb_search.py`

## 🧯 过程问题与规避策略（今天踩坑总结）

1. **PowerShell 多行续行 `>>` 容易把参数吃掉**  
   - 解决：用“变量法”写命令：`$pdf=...; python ... --pdf $pdf ...`

2. **旧脚本存在“路径/脚本不存在”问题**  
   - 典型：`scripts/251215clear_and_reindex.py` 引用 `scripts/reindex_documents.py`（项目里不存在）  
   - 解决：统一使用：
     - 单书：`scripts/reindex_single_pdf.py`
     - 批量：`python -m backend.cli.batch_indexer`

3. **图片有 URL 但仍可能“看不到图”**  
   - 原因：引用被截断/未抽取到 `images[]`/前端需要占位符才能渲染  
   - 解决：chat 侧统一抽取 `images[]` + message 占位符；同时“要图补图”

4. **图片语义检索效果依赖 VLM 覆盖面**  
   - 目前仅局部页段/限额（示例：24 张）  
   - 下一步需要扩大 VLM 覆盖，以提升“按语义命中正确平面图”的概率
