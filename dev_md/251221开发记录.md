## ✅ 今日完成情况（结论先行）

### A. `/api/v1/chat` 已支持透传检索过滤器 `filters`
- `ChatRequest` 新增 `filters: Dict[str, Any]`（可选）。  
- Chat API 在构建 `AgentRequest` 时注入 `filters`，让后续 Worker（Milvus/MongoDB）可以按过滤条件检索。  

### B. Milvus 检索已支持按资料范围过滤（doc_id / source_document）
- `MilvusChunkRetriever.search_chunks()` 新增：
  - `source_documents: List[str]`（已有）  
  - `doc_ids: List[str]`（新增）  
  并在 Milvus expr 中追加 `doc_id in [...]` / `source_document in [...]`。  
- `milvus_agent` 从 `request.filters` 读取并传入 retriever：
  - `filters.doc_ids / filters.doc_id`
  - `filters.source_documents / filters.source_document`

### C. MongoDB 关键词回退检索也支持同样的资料范围过滤
- `MongoDBChunkRetriever` 增加了“将 source_documents 反查 documents._id → 形成 chunks.doc_id $in”的解析逻辑：  
  - 支持 `doc_ids`（直接指定）  
  - 也支持 `source_documents`（用 documents.title 反查）  
  - 同时兼容 `ObjectId` 与 `str` 两种 doc_id 存储形态  
- `mongodb_agent` 将 `filters.doc_ids/source_documents` 传给 `smart_keyword_search()`，确保在“没有 chunk_ids 线索、只能关键词回退”时也不会跨资料跑偏。  

### D. 测试与排障 CLI 已就绪（全部可用 `python -m`）
- `python -m backend.cli.test_chat_images` 新增参数：
  - `--doc-id <id>`（可重复）  
  - `--source-document <title>`（可重复）  
  会在请求体中带上 `filters`，用于单资料/多资料的严格回归。  
- `python -m backend.cli.audit_mongo_doc`：可快速审计某 `doc_id` 的 chunk 结构、图片 chunk、VLM 覆盖情况，并可带 `--base-url` 输出可点击的图片 URL（便于人工验证目标页/目标图是否存在）。  

### E. PowerShell 侧的“跑不通”隐患已处理（非核心但能显著降低摩擦）
- `scripts/test_chat_images.ps1` 已转为 **UTF-8 with BOM**（内容不改），避免 PowerShell 5.1 在默认编码下解析中文脚本时报错/乱码。  
- 但总体方向仍建议：**后续回归尽量用 Python CLI**（同一套参数体系，不受 shell 编码影响）。

### 2) 仍需要提升/正在暴露的问题
1. **资料漂移（最关键）**：当用户明确要求某一资料（如《医院建筑设计指南》）时，系统仍可能引用其它资料（如《资料集6》）来“补答案”。  
   - 本次已补齐“资料范围限定”能力，但仍需要你重启服务并跑回归，确认效果与边界。  
2. **图片命中与选图不稳定**：目标页的图片 chunk 存在于 Mongo，但在 `images[]` 中不一定能稳定进入 Top（常见原因：VLM 覆盖不足、图片语义弱、排序偏向文本/其它页）。  
3. **跨资料交叉验证“结构化输出”仍偏弱**：目前更多是“混合引用”，而不是“逐条对照：一致/差异/仅见于某资料”。这需要在 Synthesizer 层做更严格的输出模板（以及配合 doc scoping 做 A/B 测试）。

### Step 1：验证“指定资料不漂移”（单资料强约束回归）
目标：确认加了 `--doc-id` 后，`citations[].source` 只来自目标资料。
```powershell
# 例：仅在《医院建筑设计指南》内回答（doc_id 以你的库为准）
python -m backend.cli.test_chat_images --doc-id 6944bd320e5acfd26a8509cc --show-citations 15 --message "请给我返回《医院建筑设计指南》里关于护理单元（病区）平面组织的典型做法（如有图请带图），并说明护士站位置与医患流线策略（引用）。"
```
检查点：
1) `citations` 是否只剩这本书（不再混入《资料集6》）  
2) 若答案变“空/弱”，用 Step 1.1 排查“资料内是否真的有对应 chunk”  

#### Step 1.1：资料内命中排查（Mongo 审计）
```powershell
python -m backend.cli.audit_mongo_doc --doc-id 6944bd320e5acfd26a8509cc --search "护理单元" --limit 20 --base-url http://127.0.0.1:8000
python -m backend.cli.audit_mongo_doc --doc-id 6944bd320e5acfd26a8509cc --search "护士站" --limit 20 --base-url http://127.0.0.1:8000
```
结果解释：
- **能搜到**：但 chat 仍弱 → 下一步调 `top_k`/提示词/检索策略  
- **搜不到**：说明该资料 OCR/分块里确实缺内容 → 下一步需要补数据或换关键词/补索引  

### Step 2：做“跨资料交叉验证”（双资料严格范围）
目标：同一问题，只允许在 A+B 两份资料里检索，逼出“对照式引用”。
```powershell
# 例：政策文件 + 资料集6
python -m backend.cli.test_chat_images --doc-id 6944bd500e5acfd26a850cd4 --doc-id 6946a9e801a4654f760b5fdf --show-citations 25 --message "仅基于这两份资料，对国家区域医疗中心/国家医学中心的布局与建设要求做交叉验证：先列出两份资料一致的要点（逐条引用），再列出差异/补充要点（逐条引用，并标注‘仅见于哪份资料’）。"
```
检查点：
- `citations` 的来源是否仅在两份资料内  
- 输出是否已经具备“逐条对照”的结构（若没有，需要在 Synthesizer 增强模板）  

### Step 3：提升“要图”问题的稳定性（先在单资料内把目标图打稳）
目标：限定《资料集6》后，目标页（如 152 页）图片能更稳定进 `images[]`。
```powershell
python -m backend.cli.test_chat_images --doc-id 6946a9e801a4654f760b5fdf --top-k 20 --only-image-citations --show-citations 30 --message "请给我返回《建筑设计资料集 第6册 医疗》里设有中毒科医院医技楼平面图（带图），并说明污染/清洁流线（引用）。"
```
若仍不稳定：
1) 先用审计确认目标图片 chunk 是否存在且 page/section/caption 是否正确：  
```powershell
python -m backend.cli.audit_mongo_doc --doc-id 6946a9e801a4654f760b5fdf --search "中毒科" --limit 30 --base-url http://127.0.0.1:8000
```
2) 再考虑扩大关键页段的 VLM 覆盖，让图片 chunk 语义更强（成本/耗时更高）：  
